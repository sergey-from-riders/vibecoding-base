# 13. Модель Развертывания И Альтернативы

Этот документ отвечает на вопрос: «Что именно поставляем на сервер и есть ли варианты лучше?».

## 1) Принятая модель для Go Next Postgres Framework

Используем **image-based deployment**:
- в CI собираются Docker-образы `api` и `web`;
- образы публикуются в registry;
- на сервере выполняется `docker compose pull && docker compose up -d`;
- Nginx проксирует:
  - `127.0.0.1:18030` (web)
  - `127.0.0.1:18031` (api)

Это означает: на сервер поставляются **контейнерные образы**, а не «ручная сборка кода».

## 2) Почему именно так сейчас

1. Повторяемость окружения между stage/prod.
2. Предсказуемый rollback по тегу образа.
3. Быстрое обновление на одном сервере без оркестратора.
4. Хорошая совместимость с текущим scope (`Auth + Switch Company`).

## 3) Варианты развертывания

### Вариант A (текущий, рекомендован сейчас): Docker Compose + Nginx
- Когда подходит:
  - 1-3 сервера;
  - несколько сервисов;
  - нужен простой и контролируемый production.
- Плюсы:
  - низкая сложность эксплуатации;
  - быстрый старт;
  - адекватный rollback.
- Минусы:
  - нет полноценного orchestration уровня Kubernetes.

### Вариант B: systemd + бинарники (без Docker)
- Когда подходит:
  - очень простой backend;
  - команда не хочет контейнеризацию.
- Плюсы:
  - минимум слоев;
  - иногда ниже overhead.
- Минусы:
  - сложнее обеспечить полную повторяемость;
  - сложнее масштабировать и переносить окружение.

### Вариант C: Kubernetes (managed/self-hosted)
- Когда подходит:
  - много сервисов;
  - высокая нагрузка;
  - несколько окружений и команд;
  - требования к HA и сложному autoscaling.
- Плюсы:
  - сильный orchestration и масштабируемость.
- Минусы:
  - высокая операционная сложность;
  - избыточно для текущего этапа.

## 4) Что «лучше» для нас сейчас

Для текущей стадии проекта **лучший баланс**:
- Docker Compose + Nginx + image-based CI/CD.

Переход на Kubernetes имеет смысл только после роста числа сервисов и требований к HA.

## 5) Практический стандарт

1. CI сначала выполняет `verify-backend`, `verify-frontend`, `verify-db`.
2. Только после успешного verify CI собирает и публикует образы по тегу коммита.
3. В `infra/docker/.env` фиксируются конкретные теги образов.
4. Сервер делает `pull` и `up -d`.
5. После деплоя запускается post-deploy smoke, включая auth/session smoke с universal test account.
6. При проблеме откат — возврат к предыдущим тегам.

Политика test gates: `docs/17-TESTING-TDD-QUALITY-GATES.md`.
